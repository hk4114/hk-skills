Perplexity 的前沿智能体产品，建立在一套模块化 [Agent Skills](https://agentskills.io/home) 之上。这里的 Skill 可以理解为“智能体技能”或“能力包”：它把实践知识和领域专业知识封装起来，供模型在需要时加载和使用。我们在不同技术环境中维护了一套经过严格筛选的 Skill 库。这些 Skill 既包括支撑 [Perplexity Computer](https://www.perplexity.ai/products/computer) 的通用工具，也包括金融、法律、健康等垂直领域能力，还包括大量用于处理用户长尾需求的模块。有些 Skill 很少被调用，但一旦被调用就非常关键。为了保证用户体验始终足够好，Perplexity 的 Agents 团队把 Skill 质量看得和代码质量一样重要。

开发高质量 Skill 所需要的直觉和最佳实践，和开发传统软件明显不同。Agents 团队会评审许多优秀工程师在工作中提交的 Skill PR。结果几乎总是会留下大量评论和修改建议。原因很简单：很多写代码时有用的模式，放到 Skill 创作里会变成反模式。

比如，如果你拿 [PEP20 - Python 之禅](https://peps.python.org/pep-0020/) 里的格言来对照，很快就会发现：写好 Python 代码和写好 Skill 不是一回事。在这 20 条智慧中，至少有一半放到 Skill 写作里是完全错误的，甚至会误导你。下面是其中五条：

| **Python 之禅** | **Skill 之禅** |
| --- | --- |
| 简单优于复杂 | Skill 是一个文件夹，不是一个文件。复杂性就是特性。 |
| 显式优于隐式 | 激活靠隐式模式匹配。采用渐进式披露。 |
| 稀疏优于密集 | 上下文很贵。每个 token 都要有最大信号量。 |
| 特例不足以特殊到打破规则 | 易踩坑本来就是特例，而且是最高价值的内容。 |
| 如果实现容易解释，它也许是个好主意 | 如果它容易解释，模型已经知道了。删掉。 |

这份指南是 Perplexity 内部工程师开发和评审 Skill 时使用的文档。我们也把它公开发布出来，希望这些发现和经验能帮助更广泛的社区。无论你是在日常工作中设计生产级 Skill 的工程师，还是想把自己最熟悉的领域做成 Skill 的 Computer 用户，或者两者都是，这份指南都适合你。

## 什么是 Skill？

当你写 Skill 时，你写的不是普通软件，尽管 Skill 现在已经是智能体系统主要逻辑引擎的一部分。更准确地说，你是在为模型及其运行环境构建上下文。Skill 有不同的约束，也有不同的设计原则。如果你像写代码一样写 Skill，你会失败。

至少在 Perplexity 的构建方式中，一个 Skill 至少同时是四件事。

### Skill 是一个目录

Skill 不只是一个单独的 `SKILL.md` 文件。很多时候，一个 Skill 会包含多个文件。在以该 Skill 命名的目录下，你可能会看到：

- `SKILL.md`：frontmatter（文件头元数据）和指令
- `scripts/`：智能体直接运行的代码，而不是每次重新发明的代码
- `references/`：体量较大的文档，按条件加载
- `assets/`：模板、schema 和数据
- `config.json`：首次运行时的用户设置

这种 hub-and-spoke（中心辐射式）模式能让 Skill 保持聚焦和紧凑，同时也给文件夹结构留下了很大的创造空间。有些特别复杂的 Skill 会受益于多层级目录，因为它能帮助模型更好地导航。假设某个 Skill 需要覆盖 300 个主题，而这些主题可以归入 20 个专业领域。即使对今天最强的前沿模型来说，要在 300 个主题里稳定选中正确主题，仍然是一个未解决的问题。让模型先锁定 20 个领域之一，再从该领域内的 15 个主题中选择，要容易得多。

举一个多层级结构带来价值的例子：在刚过去的报税季，我们团队在支撑 Computer 美国个人所得税能力的 Skill 中使用了三层主题嵌套。考虑到税法的复杂性，这种层级结构绝对不可或缺。在早期测试中，如果把美国《国内税收法典》的全部 1,945 个章节放进同一个文件夹展示给模型，效果甚至比完全不加载这个 Skill 还差。把信息组织进合乎逻辑的子分区，对于保证高精度读取至关重要。

但这种层级结构不是免费的。层级越多，信息架构上需要维护的间接关系就越多，也就越需要策划和治理。我们设计了快速参考指南、自定义搜索工具以及其他工具，帮助模型用尽量少的跳转找到信息。在这个案例中，认真做信息策划最终带来了正向结果：这个 Skill 让模型在处理税务相关任务时，比只使用通用工具强得多。

### Skill 是一种格式

Skill 是一种格式。根部核心文件 `SKILL.md` 必须同时包含名称和描述。而且，Skill 名称必须精确对应它所在的目录名。名称必须全部小写，不能有空格，可以使用连字符。description 是路由触发器。这是一个常见失败点：description 不是解释这个 Skill 做什么的内部文档。它本质上是在告诉模型什么时候应该加载这个 Skill。所以你经常会看到“Load when...”，而不是“This Skill does...”。这一点很重要，因为多数实现都会把 description 注入模型上下文。

frontmatter 里还有 `depends:`，它允许你创建层级化的 Skill 依赖；还有 `metadata:`，用于评审和评测。不同智能体系统甚至可以定义自己的 frontmatter 字段，以适配各自系统的特殊用法。另一种做法是把 Skill 专用元数据放进辅助 JSON 或 YAML 配置文件。当你构建的智能体系统需要让不同 Skill 触发不同运行时行为，同时又不想把这些细枝末节污染模型上下文时，这种做法更合适。最后，也可以在读取时剥离 Skill frontmatter 来获得类似效果。Computer 使用的就是这种方法：配置仍然保存在根部 `SKILL.md` 中，但读取时可以把它从模型可见内容里移除。这要求解析逻辑足够细致；如果某些字段对模型上下文确实有用，你可能还需要实现条件式剥离。

### Skill 是可调用的

Skill 是可调用的。智能体会在运行时加载 Skill。重点是，Skill 并不总是被打包进上下文。默认情况下，多数智能体系统会按具体需求逐步展开 Skill。

按照我们在 Computer 里的实现方式，Skill 至少有三层上下文成本。流程如下：

1. Computer 调用 `load_skill(name="...")`
2. Computer 把 Skill 目录复制进隔离的执行沙箱
3. Computer 递归自动加载 `depends:` 标签中的依赖
4. Computer 随后剥离 frontmatter，于是智能体只看到正文和附加文件

不同智能体系统可以用不同方式暴露 Skill 内容。例如，有些系统可能完全不暴露文件层级，而是让模型通过文件系统操作自己发现层级。另一些系统可能会给模型一份完整文件树映射，只是加上截断或深度限制。为了保持上下文干净，Computer 默认不会在调用上下文中放入完整文件层级；不过这个行为可以按单个 Skill 覆盖。

### Skill 是渐进式的

Skill 是渐进式的。在 Computer 中，上下文成本分成三层，我们会在不同阶段支付这些成本：

| **层级** | **加载内容** | **预算** | **什么时候支付** |
| --- | --- | --- | --- |
| 索引 | 每个非隐藏 Skill 的 `name: description` | 每个 Skill 约 100 token | 每个会话、每个用户、始终支付 |
| 加载 | 完整 `SKILL.md` 正文 | 约 5,000 token | 约 5,000 token |
| 运行时 | `scripts/`、`references/`、`assets/`、subskills、`FORMATTING.md`、`SPECIAL_CASES.md` 中的文件 | 无上限 | 只有智能体读取它们时支付 |

Computer 会构建一个 Skill 索引，其中包含每个可用 Skill 的名称和描述。这个预算大约是每个 Skill 100 个 token，越短越好。它之所以这么紧，是因为你要在每个会话、每个用户身上支付这笔成本。它会在对话一开始被注入系统提示词。模型可以看到一组具名 Skill 和描述，从而判断是否调用 `load_skill()`。能进入这个索引的门槛极高。你的 Skill 必须非常有用，而 description 必须极其高密度、极其简短，因为所有用户一直都在为它付费。

智能体系统加载 Skill 之后，才会出现完整的 `SKILL.md` 正文。理想情况下，正文不要超过 5,000 个 token。即便如此，你仍然希望每句话都有价值，因为一旦加载了某个 Skill，接下来整段对话都会背着这部分上下文，直到到达压缩边界。很多线程会加载三到五个不同 Skill，这个成本会被放大。充满废话的 Skill 几乎肯定会拖累其他 Skill，也会拖累整体智能体能力。直白说，如果你的 Skill 被加载了，但没有做对事情，那就是浪费上下文。

最后一层渐进性来自脚本或特殊情况，比如 subskills 或格式规则。无上限的条件分支逻辑应该放在这里。智能体只有在需要时才会使用它，所以这里可以容纳的内容门槛低得多。

在索引层，每个 token 都重要。加载后的 Skill 正文可以稍微宽松一些。运行时层最宽松，可能是 20,000 个 token，也可能是 0 个 token。这一层才是你应该考虑如何渐进式扩展模型上下文的地方。

## 什么时候需要 Skill？

Agents 团队经常被问到：某个领域或用例到底是否真的需要 Skill。仅凭第一性原理，我们很少能给出确定答案。真正弄清楚的方法只有一个：先让你的智能体在没有这个 Skill 的情况下运行几个代表性查询（hero queries，最能暴露能力边界的核心测试问题），然后判断它做得好不好。

### 什么时候需要 Skill

很多任务本来就在训练后模型的能力分布内。只有当你想以某种具体方式改变这种行为，而且无法通过提示词里的一句话做到时，才需要使用 Skill。也就是说，当没有特殊上下文智能体就会做错，或者存在某种你必须跨运行保持高度一致的不一致性或非确定性时，你才需要 Skill。

原因可能是你的知识很稳定，但没有进入训练数据；也可能是存在知识截止日期、企业专用工作流，或者只是品味问题。例如，Computer 中有几个设计相关 Skill 是 Henry Modisett（我们的设计负责人）写的。这些 Skill 里每个 token 都存在，是因为 Henry 在网站和 PDF 设计上有非常好的品味。Henry 会明确哪些字体该用、哪些字体不该用、这些字体给人的“感觉”是什么，以及其他模型无法仅靠训练数据学到的判断。

### 什么时候不需要 Skill

我们见过很多 Skill，里面写了一系列需要按顺序执行的 git 命令。这没有必要，因为模型已经知道怎么做。这类内容可以是很好的文档，但不是好的 Skill。

我们也见过一些 Skill 只是重复系统提示词里的指令。你不需要为此写 Skill。大多数请求都会用到的知识，应该放进全局上下文，而不是放进一个条件加载的 Skill。

如果某个东西变化得比你能维护它的速度还快，你也不需要 Skill。比如，你在调用某个远程 MCP 端点，而它的工具或工具版本经常变化，那就不应该把这些信息注入 Skill。你要是这么做，最后只会产生漂移，模型也会犯错。

### 每个 Skill 都是一笔税

你可以用一个测试来审视 Skill 里的每句话：“如果没有这条指令，智能体会做错吗？”如果这句话不必存在，它就负担不起存在的成本，因为每个会话、每个用户都在为它付费。当你判断是否要新增一个 Skill 时，要记住这笔税：它会让每个会话和每个用户都消耗 token。

下面这句名言用法语说要好听得多，大意是：“我只是因为没有时间把这封信写短，才把它写得这么长。”

> « Je n’ai fait celle-ci plus longue que parce que je n’ai pas eu le loisir de la faire plus courte. » — Blaise Pascal, Lettres Provinciales, 1657

和 Pascal 一样，你也需要在每个 Skill 上投入时间。写短 Skill 很难。如果你的 Skill 很容易写出来，它大概率太长，或者根本不该存在。好的 Skill 已经短到不能再短。

如果你发现自己在一次性生成 Skill，然后五分钟内就提交 PR，结果几乎一定不够好。事实上，早期研究已经[显示](https://arxiv.org/abs/2602.12670)，如果你用 LLM 来写 Skill，LLM 很可能不会从中受益：“自生成 Skill 平均没有收益，这说明模型无法可靠地编写自己在使用时会受益的程序性知识。”

## 如何构建 Skill

换句话说，你必须把自己的判断注入你写的每个 Skill。按下面这些步骤来。

### 第 0 步：先写评测

先写一些评测。评测用例可以来自：

- 真实用户查询：从生产环境或你的智囊团中抽样
- 已知失败：因为没有 Skill，智能体失败过
- 邻近混淆：靠近你的领域边界，但容易路由到另一个 Skill 的请求

至少，你要确保自己在测试“该加载 Skill 时是否真的会加载”。理想情况下，你可以从生产环境抽一些样本。你也应该考虑已知错误案例：也许你之所以决定写这个 Skill，就是因为观察到了某个具体失败；又或者你正在重构，而两个相近领域分别由同一个 Skill 覆盖，出现了混淆。

从相似的负例和正例开始。负例极其强大，而且可能比正例更重要。

### 第 1 步：写 description

这是整个 Skill 里最难的一行。它是路由触发器，不是文档。为了写对名称和 description，你并不关心 Skill 的内容是什么。你只关心它是否会在正确时机被加载和注入，以及是否不会产生偏离目标的副作用。这种副作用是头号失败模式。每增加一个 Skill，你都可能让其他每个 Skill 轻微变差，所以必须尽量减少回归。

再次强调，糟糕的 description 会描述这个 Skill 做什么，或者为什么有用。好的 description 会说明智能体什么时候应该加载它。比如，你有一个用于监控 PR 的 Skill。不要写它做什么。要写工程师在焦虑时真正会说的话，比如 “babysit”“watch CI”“make sure this lands”。

这里有一份快速检查清单：

- 以 “Load when...” 开头
- 目标控制在 50 个词以内
- 描述用户意图，最好来自真实查询
- 不总结工作流

真实查询通常就是你能覆盖的 80/20。一般来说，两三个例子效果就很好。做到“刚好只加你需要的内容”，并不容易。

### 第 2 步：写正文

接下来，写 Skill 本身的内容。注意，这不是第 0 步，也不是第 1 步。

向 LLM 传达工作流，和向同事甚至运行时系统传达工作流完全不同。当工程师学习一个新的软件工具时，他可能需要读文档、听有经验的人讲一遍、再学会怎么用这个工具。但对于几乎任何已经存在至少一年的软件工具，你只需要提到它的名字，LLM 就已经知道足够多的信息。

写正文时，要跳过显而易见的内容。很多工程师都很熟悉写 `README.md`：把某人需要运行的每条命令都列出来。写 Skill 时很容易退回这种习惯，因为它看起来像是在写文档。但如果你这么做，你的 Skill 会很糟糕。所以，不要写一串命令。

比如，你不需要写：

`git log # find the commit; git checkout main; git checkout -b <clean-branch>; git cherry-pick <commit>;`

应该写：

“把这个提交 cherry-pick 到一个干净分支上。解决冲突时保留原意。如果无法干净落地，解释原因。”

后者会让模型表现好得多，尤其是在事情出错时。不要把模型钉死在过度规定的步骤上，因为那很脆弱；在多种方法都可行时，保持灵活。再说一次，给人类看的好文档，很多时候是给模型看的坏文档。

接下来，关注易踩坑或负例。这些内容信号密度极高，因为它们经常是在告诉模型不要做什么。每当智能体跌倒一次，你就加一条，运行久了，这些 gotchas 会自然增长。

最后，如果某一部分是条件性的，或者内容极重，就把它从作为中心的 `SKILL.md` 中拿出去，放进某个 spoke（辐射文件）里。把它放进一个可渐进加载的附属文件。下一节会继续讲这一点。

### 第 3 步：使用层级结构

当你有脚本、参考资料，或者要使用某个特定工具时，要用好 Skill 层级：

| `scripts/`  智能体每次都会重新发明的确定性逻辑 | 给它可组合的代码，而不是让它重建 |
| --- | --- |
| `references/`  只在某个条件满足时加载的重型文档 | “如果 API 返回非 200，读取 `api-errors.md`” |
| `assets/`  智能体复制并填充的输出模板 | `report-template.md`、输出 schema |
| `config.json`  首次运行时的用户设置 | 询问 Slack 频道，保存，下次复用 |

任何条件性或分支性的内容，都应该从主 Skill 中拆出去，放到文件夹里。还要记住，多层级结构也适用于特别复杂的 Skill。对这类 Skill，你需要仔细思考：它应该做成单体功能，还是做成一组 Skill，可能再通过 `depends:` 建立加载关系。

### 第 4 步：迭代

接下来，在一个分支上做大量迭代。从没有 Skill 的 main 分支开始，跑几轮，建立你的代表性查询集，然后运行大量评测。任何评审你 Skill 代码的人，都会感谢你提交的是一个包含完整评测集的单一 changeset。连续评审多个增量改动非常困难，除非只是新增 gotcha，所以尽量减少这种情况。

你很可能会做很多小的措辞修改。description 里的小措辞变化，可能对路由产生巨大影响，包括对其他 Skill 的溢出影响。因此，要在第 5 步之前把这些工作做完。

### 第 5 步：发布

发布它。

## 如何维护 Skill

写完 Skill 之后，你还要维护它。

### Gotchas 飞轮

从这一刻开始，你的 gotchas 列表往往会频繁增长和变化。我们经常看到工程师提交未经评测的 PR，比如修改 description。如果你的 Skill 已经合并之后还在改 description，那你就偏离了轨道。只要你在改决定 Skill 是否路由的东西，就必须写评测来支撑这些改动。

Skill 以追加为主。gotchas 区域会随着时间积累最多价值：

- 智能体在某件事上失败 → 添加一个 gotcha
- 智能体在错误目标上加载 Skill → 收紧 description，并添加负向评测
- 智能体该加载 Skill 时没有加载 → 添加关键词和正向评测
- 系统提示词变化 → 检查是否存在冲突或重复

在内部测试或生产中发现一个失败案例，然后加一个 gotcha，这很容易。它是一个负例，所以并不会真正改变显式指导，但它会让模型知道：“注意，这里有一个已知失败点。”

当你从 80/20 继续推进到 99.9% 或 99.99% 成功率时，gotcha 列表很容易增长。随着这些负例出现，你应该主要把它们追加到 gotchas 区域。不应该不断添加更长的指令，也不应该频繁修改 description。

### 评测套件

在 Perplexity，我们运行很多评测套件来检查不同问题。其中包括 Skill 加载和 Skill 文件读取评测，用来检查 Skill 加载本身的精确率、召回率和禁止项。智能体会不会在应该路由到你的 Skill 时真的路由过去？这些评测能确保新 Skill 不会破坏已有边界。

还有一些评测会检查渐进式加载是否正确。智能体可能加载了 Skill，但它有没有读取相应的附属文件？例如，如果你有一个用于金融查询的 finance Skill，它是否会读取专门的 `FORMATTING.md` 文件？

还有一些 Skill 评测会测试领域内的端到端任务完成情况。我们会跑完整智能体循环，并使用 LLM 裁判根据定义清晰的评分标准给结果打分。

最后，很重要的一点是，要在不同模型上运行这些评测。Computer 至少支持三个不同的编排模型家族：GPT、Claude Opus 和 Claude Sonnet。你应该让 Skill 加载评测和领域 Skill 评测都覆盖这些不同的智能体编排器，以确保不会出现不同行为。Sonnet 和 GPT 在处理 Skill 时的行为相当不同。

### 最后的想法和要点

你构建的 Skill 越多，你就越擅长构建更多 Skill。如果你还没有开始用 Skill 自动化那些日常重复任务，应该马上开始。

构建 Skill 本身会让你更擅长构建 Skill，而且 Skill 也非常适合自动化业务流程。如果你能描述某件每周站会前、每个 sprint 结束时，或者作为工程师每天、每周、每季度都会做的事，你就应该写一个 Skill，把时间买回来。

你能自动化事故复盘吗？你能评审 PR 吗？任何你能做的任务，至少都可以让 Agent Skill 先跑第一遍。它会为你节省大量时间。

话虽如此，要记住：Skill 并不容易，也并不总是必要。少即是多。还有几个要点：

1. 先写评测，再写 Skill。包括相邻但不同 Skill 的负例和禁止加载用例。
2. description 是最难的部分。“Load when...”（每个词都会消耗注意力）。
3. Gotchas 是极高价值内容。先写薄一点，随着智能体失败再增长。

记住，新增一个 Skill 很容易破坏其他已经存在的 Skill，即使你完全没有碰它。要警惕隔空影响（action at a distance，局部改动影响远处行为）。

每次编写和维护 Skill 时，都要使用所有可用工具。如果你想了解更多，[Agent Skills](https://agentskills.io/home) 网站上有大量好例子，我们内部仓库和公共生态中也有很多设计良好的 Skill 示例。
